Về phong cách code

Mọi người đọc tham khảo, ta sẽ thực hành khi vào code

Nếu mọi người làm việc với 1 framework(đặc biết là backend) thường thì:

  1. Chia nhỏ các phần ra để dễ kiểm soát (controller,service,view ,..bla..bla)
  2. Giờ thử đọc code của mấy ông làm lâu năm xem,Ta sẽ toàn nhìn thấy hàm trong interface việc tìm class(implements) chứa logic code cực khó.
  3. Đến Spring nó cũng chỉ cho ta xem interface và ta chỉ có kế thừa lại và dùng
  4. Trong khoá này và những khoá về sau(nếu có) mình sẽ hướng mọi người phong cách code tương tự vậy
  5. Tóm lại, từ chương này về sau ta sẽ code theo kiểu ném ra interface(chứa hàm với mục đích riêng ...) logic code sẽ để ở nơi khác
  6. Chắc chắn 1 điều, với tư duy code này ae out trình hết mấy đứa cùng trường :v.

Yếu tố thiết yếu

Không có khái niệm đâu là code tối ưu nhất với dự án kể cả với các ngôn ngữ khác (JS,PHP,GoLang,..)

Bỏ tư duy code chỉ cần hiểu , mà phải code sao cho phục vụ và giải quyết tốt bài toán của chúng ta.=> đến lúc đấy tự hiểu :v

    Hãy luôn ưu tiên code theo 3 yếu tố:
  1. Code phải có khả năng mở rộng(phát triển trong tương lai)
  2. Phải bảo trì được(chải ai biết sau này có bug hay không ,..code lỗi xong vứt là dở :v)
  3. Code phải có khả năng test(viết testcase - thường thì học cao mới thở được)

Cách code với implement

Đọc là biết là ta sẽ code như thế nào rồi nhé :v

  1. Ở đây chúng ta viết logic ở phần Service. => chuyển tất cả class trong package serivce thành interface
  2. Tạo thêm package impl(impliment) kế thừa lại những interface đó
  3. Code sẽ dài ra.Đúng . Nhưng sẽ đảm bảo tính ràng buộc khi kế thừa lại:Lỗi cái báo luôn => dễ fix bug.
  4. Ông nào lười thì code kiểu cũ cũng được :v
  5. Cuối cùng khai báo Service interface vào controller. Việc chạy logic Spring lo hết :v .

II/ . 1 số lưu ý trước khi viết api trên controller:

  1. Tham số nhận vào (request) phải là dạng JSON(Không phải truyền nguyên cái class của java đâu nhé :v)
  2. Ta sẽ dùng anotation @RequestBody trước khi khai báo tham só nhận vào.
  3. Ta sẽ không render view đâu mà ta sẽ trả về (responese) JSON
  4. Đặt anotation @ResponseBody vào đầu enpoint để Spring trả về JSON thay vì render view
  5. api có thể giống nhau nhưng method(get,put,patch,delete) khác nhau thì sẽ hoạt động trả về kết quả khác nhau
  6. Việc còn lại Spring lo hết.

DTO (Data Transfer Object) : tài liệu ae copy paste gg nhé

DTO (data transfer object) là cách chúng ta biến đổi object này sang object khác.

Mục đích : tăng thêm/hoặc giảm bớt các thuộc tính của class

Mapper thường được sử dụng để chuyển đổi giữa DTO (Data Transfer Object) và Entity hoặc giữa các đối tượng thuộc các lớp khác nhau.

Phải học chắc hương đối tượng (OOP)

Ví dụ khi đăng kí User sẽ có các thông tin khác như : nhập lại mật khẩu , họ , tên, tên đệm ,...Ta lại phải tạo thêm cột tương ứng với model User.

Nếu giữ tư duy tạo cột liên tục như vậy(cũng được) những chả khác gì ta lưu thừa dữ liệu không cần thiết vào database.Hay còn gọi là lưu dữ liệu rác.

Mình sẽ hướng dẫn mọi người kĩ thuật Mapper DTO để giảm thiểu lưu thừa dữ liệu.

Theo tư duy ở II/ Việc ta cần làm là :

  1. Tạo ra 2 đổi tượng (class) request(truyền lên) và response(trả về)
  2. Chuyển đổi 2 đối tượng đó theo đúng yêu cầu.
  3. Với người mới thì khi làm việc này khá là khó khăn, không sao code nhiều rồi cũng quen :v
